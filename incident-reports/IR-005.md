## Executive Summary

A simulated red team exercise was conducted against the home SOC lab environment to validate detection and response capabilities. The attacker (Kali Linux, 192.168.1.73) performed a multi-phase attack against the Windows 10 endpoint (DESKTOP-G5P06GA, 192.168.1.76), including network reconnaissance, service enumeration, and a WinRM credential stuffing attack. The full attack chain was detected by the Suricata IDS and Wazuh SIEM, with automated containment triggered within seconds of the brute force phase. All alerts were forwarded to Splunk via HTTP Event Collector (HEC) and a formal case was opened in TheHive 5 for incident tracking.

**Key Outcome:** The SOC pipeline (Suricata → Wazuh → Splunk → TheHive) successfully detected, alerted, and contained a credential attack with zero manual intervention required.

## Attack Timeline

### Phase 1 — Network Reconnaissance (04:10 UTC)
**Technique:** T1046 — Network Service Discovery

Attacker performed a SYN scan across all 65,535 ports of the target Windows endpoint.

**Open ports identified:**
| Port | Service |
|---|---|
| 5040/tcp | Unknown (Windows service) |
| 5357/tcp | WSDAPI (Web Services Discovery) |
| 5985/tcp | **WinRM / WS-Management** |
| 7680/tcp | WUDO (Windows Update Delivery Optimization) |

**Significance:** WinRM (5985) is a high-value target — it allows remote PowerShell execution and lateral movement if credentials are obtained.

---

### Phase 2 — WinRM Credential Stuffing (04:12 UTC)
**Technique:** T1110.001 — Brute Force: Password Guessing

25 credential pairs attempted against the WinRM endpoint using NTLM authentication:

- **Usernames:** admin, administrator, guest, Admin, William
- **Passwords:** password, 123456, admin, Password1, Admin123!

All attempts returned HTTP 405 (Method Not Allowed) — the endpoint does not support basic auth over HTTP, requiring NTLM/Kerberos. This is a detection-positive response confirming the attack reached the service layer.

**Suricata detection:** Rule `SURICATA HTTP Request unrecognized authorization method` fired on every attempt, generating 25+ alerts in under 3 seconds.

---

### Phase 3 — NetBIOS/SMB Enumeration (04:12 UTC)
**Technique:** T1135 — Network Share Discovery

Attacker attempted SMB share enumeration via nmap scripts (`smb-enum-shares`, `smb-enum-users`) on port 445. Port 445 was filtered — no shares exposed.

---

### Phase 4 — OS Fingerprinting (04:12 UTC)
**Technique:** T1082 — System Information Discovery

Service version detection run against open ports to identify OS and software versions. Suricata flagged the aggressive scan traffic pattern.

## Detection

### Suricata (IDS — First Line)
- **Alert:** `SURICATA HTTP Request unrecognized authorization method`
- **Trigger:** NTLM auth attempts against WinRM over plain HTTP
- **Volume:** 25+ alerts in a 3-second window
- **Source IP:** 192.168.1.73
- **Dest IP:** 192.168.1.76:5985

### Wazuh (SIEM — Second Line)
- Ingested Suricata alerts via log monitoring
- Correlated brute force volume and triggered active response policy
- **Active Response:** `firewall-drop` rule executed — source IP (192.168.1.73) blocked at iptables for 600 seconds
- **Alert rule fired:** "Host Blocked by firewall-drop Active Response"
